## Summary

| Case | Binding bottleneck | Saturation RPM | MaxRPS |
|------|--------------------|---------------|--------|
| queue-analysis | Markovian queue overload (overloaded) | 42 | — |
| blis-roofline | Decode bandwidth (pre-sim) | 14 | 0.1954 |
| blis-trained-physics | Decode bandwidth (pre-sim) | 14 | 0.1954 |

**Notes:**
- Both BLIS cases fire the analytical bandwidth check at RPM=14 (0.2333 RPS × 1024 output tokens > 195.4 TPS ceiling). `maxRPS=0.1954` = 11.7 RPM, consistent with the design spec prediction of ≈12 RPM.
- Both BLIS cases return zero DES metrics at saturation (pre-sim exit path confirmed working).
- The `maxRPS` field (0.6614) for queue-analysis reflects the Markovian service capacity; the evaluator does not use a pre-sim check — it runs to completion and flags `overloaded` when the DES shows degraded-state operation (still-queued requests).
- **Latency backend contrast:** At RPM=10, roofline shows TTFT=1524ms / ITL=3.78ms; trained-physics shows TTFT=33ms / ITL=8.73ms. Both backends reach the same saturation point (hardware-limited by bandwidth), but the trained-physics model produces substantially lower TTFT and higher ITL, reflecting the different latency model.
- **queue-analysis vs. BLIS discrepancy:** The large difference in saturation RPM (42 vs. 14) is not a correctness issue — it reflects uncalibrated queue model parameters. The Markovian perfParms used here (α=8.0ms, β=0.016ms/tok, γ=0.0005ms/tok²) were taken from a prior calibration run against a different workload point and have not been re-calibrated for the granite-3.1-8b / H100 configuration used in this sweep. A properly calibrated queue model would be expected to produce a saturation point consistent with the hardware bandwidth ceiling that BLIS computes analytically. Calibration is left as future work.
